# Review 模块技术方案

> 龙头回测验证系统 — 验证四维评分的实际表现

---

## 一、背景

`dragon-quant` 每天 scan 产出 topN 龙头的四维评分，写入 `dragons` 表。但这些龙头在后续交易日实际表现如何？评分高的票是否真的有更大机会？需要一套回测机制来量化验证。

## 二、核心逻辑

```
scan 执行时                       review 执行时（用户手动触发）
───────                          ──────────
写入 dragons                     读取 pending 记录
(trade_date + 基本面快照)          ↓
      ↓                          拉取历史日K（雪球）
5 日窗口去重                      找入选日后第一个非一字板日 → 买入价
      ↓                          先算收益窗口内最大收益 / 持有天数 n
buy_date = NULL                   ↓
review_status = 'pending'         再算买入日至峰值日窗口内最大回撤
                                   ↓
                                  UPDATE dragons
```

### 2.1 入选去重

同一标的在 **5 个交易日内** 只写入 `dragons` 一次。

- 金螳螂 5 月 6 日首次入选 → `trade_date = 2026-05-06`
- 5 月 7 日再次入选 → 距上次 1 个交易日，**跳过**
- 5 月 12 日再次入选 → 距上次 4 个交易日，**跳过**
- 5 月 13 日再次入选 → 距上次 5 个交易日，**允许写入**

实现：`orchestrator.py` 写入前调 `db.get_last_entry(code)` 获取上次 `trade_date`，用交易日历计算间隔 (<5 则跳过)。

### 2.2 交易日历

基于雪球平安银行（000001）日K构建：

```python
calendar = build_trade_calendar("2026-05-01", "2026-05-31")
# → {"2026-05-06", "2026-05-07", "2026-05-08", ...}  # 只含交易日

trade_days_between("2026-05-06", "2026-05-13", calendar)
# → 5  (05-07, 05-08, 05-11, 05-12, 05-13)
```

优点：不依赖任何外部假期表，日K天然过滤了周末和节假日。

### 2.3 买入时机

从入选日 `trade_date` **之后** 开始逐日查找，找到第一个**非一字板**交易日作为买入日。

当前实现使用的是“是否可介入”规则，而不是传统涨停判断：

```python
def find_entry_day(klines, entry_date):
    future = [k for k in klines if k["date"] > entry_date]
    future.sort(key=lambda k: k["date"])
    for k in future:
        if k["high"] != k["low"]:
            return k
    return None
```

- `high == low`：视为一字板，全天无可交易波动，跳过
- `high != low`：视为可介入，`buy_price = 该日日K最低价`

如果后续都找不到可介入日 → `review_status = 'no_entry'`。

### 2.4 收益/回撤回测

当前实现的收益观察窗口并不是“买入后连续 5 个交易日”直取，而是：

1. 构造买入日之后的交易日历
2. 跳过买入日
3. 再跳过买入后的第 1 个交易日（A 股 T+1）
4. 取第 2~5 个交易日作为收益观察窗口（代码上最多 4 天）

即：

```python
after_buy = sorted([d for d in calendar if d > buy_date])
future_dates = after_buy[1:5]  # 第 2~5 个交易日，共 4 天
```

在该窗口内先计算：

```python
peak_k = max(future_klines, key=lambda k: k["high"])
max_return_5d = (peak_k["high"] - buy_price) / buy_price * 100
max_return_hold_days = 交易日历中 buy_date < d <= peak_date 的交易日数
```

然后最大回撤改为只统计“买入日至最大收益出现日”窗口：

```python
drawdown_klines = [k for k in klines if buy_date <= k["date"] <= peak_date]
max_drawdown_5d = (min(low) - buy_price) / buy_price * 100
```

因此当前 `max_drawdown_5d` 的语义是：

- **买入日至峰值日之间，相对买入价的最大浮亏**
- **不包含峰值日之后出现的更低低点**

## 三、DB Schema

`dragons` 表新增/维护 6 列：

| 列名 | 类型 | 说明 |
|------|------|------|
| `buy_date` | TEXT | 买入日 YYYY-MM-DD |
| `buy_price` | REAL | 买入价（断板日日K最低价） |
| `max_return_5d` | REAL | 收益观察窗口内最大收益率 (%) |
| `max_drawdown_5d` | REAL | 买入日至峰值日窗口内最大回撤率 (%) |
| `max_return_hold_days` | INTEGER | 达到最大收益的交易日数 |
| `review_status` | TEXT | pending / completed / no_entry / error |

新增索引：`idx_dragons_review ON dragons(review_status, trade_date)`

迁移策略：幂等 `ALTER TABLE ADD COLUMN`（已有 `_ensure_schema` 中的迁移模式）。

## 四、CLI 命令

```bash
# 回测全部待处理记录
dragon-quant review

# 指定日期
dragon-quant review --date 20260519

# 只回测 top 5
dragon-quant review --top 5

# 强制重算（无视 review_status）
dragon-quant review --force --date 20260519

# 回测完成后启动 Web UI
dragon-quant review --ui

# 仅启动 Web UI
dragon-quant review --ui-only --port 8765
```

## 六、边界情况

| 场景 | 处理 |
|------|------|
| 入选后持续一字板 / 无可介入日 | `review_status = 'no_entry'` |
| review 时未来窗口数据不足 | 只算已有数据；若没有未来窗口则收益/回撤返回 0 |
| `--force` 强制重算 | 调 `get_pending_dragons(..., review_status=None)`，支持无视状态全量重算 |
| 002 / 北交所 30cm 涨停 | 入选前已过滤（`_is_valid_candidate`），不进入 dragons |
| 停牌期间 | 日K无数据，`find_entry_day` 自动跳过无K日期 |
| 买入当天仍强势 | 只要 `high != low` 即认为存在可介入波动，买入价取当日 low |

## 七、评审结论

- **回测操作是异步的**：系统不自动勾采数据，自己 run review
- **评分和回测结果独立存放**：各维度评分在 `scan_stocks` 中，回测结果在 `dragons` 中
- **盈亏按短窗口验证**：最大收益取收益观察窗口，最大回撤只看买入日至峰值日，不做长期持仓评估
